1. Project overview

ControlShowcase is a desktop application for learning control systems by watching them run. It simulates a bidirectional hydraulic cylinder whose position is driven by a PWM electronic proportional valve, and lets you control that cylinder several ways — by hand, with a PID controller, or with a model-predictive controller (MPC) — so the behaviour of each can be seen and compared on the very same plant.

The point is pedagogical. A manual mode builds intuition for the open-loop plant; a PID mode with live P / I / D sliders makes the effect of each term visible in real time; an MPC mode — written from scratch in plain numpy so the maths is legible — invites a direct comparison with PID. A final auto-calibration step discovers the valve's deadband and proposes a reasonable PID tuning automatically.

It is the control-systems sibling of CsvPlotter and follows the same conventions: PySide6 + pyqtgraph, a hand-coded dark UI, a three-layer architecture, and this index.html + shared style.css as a living document with a phased roadmap. Its CSV export (a later phase) feeds straight back into CsvPlotter.

2. Current phase Phase 5 — auto-calibration

The last control mode is in: auto-calibration. Rather than Ziegler–Nichols or relay methods — which deliberately push the loop to sustained oscillation to find the stability limit, risky on a heavy axis — it uses a safe, model-based approach. A short, scripted sequence of gentle, bounded, open-loop moves identifies the valve deadband and a simple plant model (velocity gain + lag); the PID gains then follow analytically from SIMC / lambda rules with a single aggressiveness knob. Nothing ever approaches instability. It runs live (you watch the moves), reports the findings, and an Apply button loads the gains into the PID tab.

All four modes — Manual, PID, MPC, Calibration — are now live. The headline comparisons hold: PID vs. MPC under a known load, and now a one-click route from an untuned plant to a working PID baseline. Phase 6 (polish: run overlays, CSV export, presets) remains.

Why numpy-only for MPC? The teaching value is in seeing the prediction and the cost function, not in a black-box solver. A small receding-horizon problem with box constraints is solved directly in numpy, so every step — the prediction matrices, the tracking/effort cost, the clamp to the PWM limits — is right there in the code. No cvxpy, no scipy.

5. How it works (architecture)

The design uses the same three-layer separation as CsvPlotter, with a control layer added between the plant and the UI:

  • The plant (sim/) — the physics. A Valve turns a PWM command into a spool position; a Plant (the cylinder) turns that into motion; a Simulator owns the state, advances it one fixed step at a time, and keeps a rolling history of (t, setpoint, position, pwm, error). No Qt here at all.
  • The controllers (control/) — each implements the same tiny interface, compute(setpoint, measurement, dt) → pwm: manual passthrough, PID, MPC, and the calibration routine (a state machine that scripts its own bounded moves). They are pure functions of their inputs and their own state; they don't know about Qt or the simulator.
  • The UI (ui/) — the control panel emits signals when parameters or the mode change; the cylinder view and the plot are told what to draw. Widgets never mutate the plant.
  • MainWindow — the glue. It owns the Simulator and the active Controller, runs a QTimer, and on each tick does pwm = controller.compute(…) → simulator.step(pwm, dt) → refresh the cylinder view and the plot.

The single source of truth is the Simulator's state and history; everything the user sees is a view of it, the same discipline CsvPlotter uses with its Series model.

6. The plant model

The cylinder is simulated as a second-order system with realistic nonlinearities, which is exactly what makes the controllers interesting to compare. Implemented in sim/ across three small, Qt-free modules — valve.py, plant.py, simulator.py.

The valve

Input is a PWM command in [-100, +100]. It passes through a deadband (no flow until the command exceeds a threshold), a saturation (clamped to ±100), and a first-order lag (the spool can't move instantly): s += (eff - s) · dt/τ.

The cylinder

State is position x and velocity v. The spool position commands a flow, which sets a target velocity with asymmetric gains for extend vs. retract (the bore and rod sides have different areas). Net force subtracts viscous and Coulomb friction and an adjustable external load (a weight on the rod that opposes extension, set 0–100% from the panel); the result is integrated to velocity then position, with end-stops that clamp the stroke. Deadband, asymmetry, saturation, friction, and the load together are why a single fixed PID can't be perfect everywhere — and why MPC's preview (and, later, a known-load feed-forward) can help.

The simulator

Simulator owns one Valve and one Plant, the current setpoint, and a rolling history of (t, setpoint, position, pwm, error) (a fixed-length deque, exposed as numpy arrays for the plot). Each step(pwm) advances the valve then the plant by one fixed step (default 200 Hz) and records a sample — this is the single source of truth the UI reads. Default plant: 500 mm stroke, ~180 mm/s extend vs. ~240 mm/s retract, 8% valve deadband.

Sensor noise. Each step records a measured position — the true position plus configurable Gaussian noise (std-dev in mm, set in the control panel; 0 = perfect sensor) — and the plot's blue trace shows it, so you can see how noisy the reading is. The noise is applied to the displayed position only: the controllers are driven by the true position, so a noisy sensor does not jitter the PWM command, and the cylinder animation always shows the true position. The sample is recorded before integrating forward, so measured and true position stay aligned (identical when noise is off).

Run python -m controlshowcase.sim.simulator for the headless sanity checks (deadband leak, open-loop ramp, extend-vs-retract asymmetry, end-stop saturation) plus a printed step response.

7. The controllers

Every mode is a Controller with one method, compute(setpoint, measurement, dt) → pwm, and the MainWindow runs whichever the active tab selects on each 200 Hz step. Manual is the trivial case — it returns the slider value and ignores the setpoint. The three real controllers follow.

PID

Classic position PID on the error e = setpoint − measured, with the practical refinements that make it usable on this plant:

  • Output clamping to ±100% PWM.
  • Anti-windup by conditional integration — the integral only accumulates when doing so wouldn't push an already-saturated output deeper into saturation, so sitting far from the setpoint doesn't wind up a big overshoot.
  • Derivative on the measurement, low-pass filtered — differentiating the position (not the error) avoids a "derivative kick" when the setpoint steps, and the filter smooths the D term's high-frequency response.

Units: kp is %/mm, ki %/(mm·s), kd %/(mm/s). The integral is what removes the steady-state error left by the valve deadband and a constant load — set I to zero and the droop returns. Gains are editable live; the integral and filtered derivative carry across edits so tuning feels continuous.

MPC (model-predictive control)

Standard linear MPC, hand-rolled in numpy. Each control instant it predicts the cylinder's motion over a short horizon, picks the PWM sequence that minimises a cost, applies only the first move, then re-solves on the next step (receding horizon).

  • Prediction model — the plant's linear core (state [position, velocity], input PWM, the load as a known constant force), discretised exactly with a zero-order hold. The nonlinearities (deadband, asymmetry, Coulomb friction) are deliberately left out; re-solving every step keeps it on track despite the mismatch.
  • Cost — Q·‖position − setpoint‖² + R·‖ΔPWM‖² over the horizon, subject to −100 ≤ PWM ≤ 100. It penalises the change in PWM (move suppression) rather than its magnitude, so holding a large steady command (e.g. against a load) is free — essential for the feed-forward below to hold instead of drooping.
  • Solver — the condensed box-constrained QP is solved with a FISTA-accelerated projected-gradient method, warm-started from the previous solution. Small, fast, no external solver.
  • Known-load feed-forward — because the load is in the prediction model, the optimiser already knows it must hold against the weight and commands the PWM that does. This is the headline contrast with PID, which only learns of the load once the error appears. (The app tells the MPC the load; it does not tell the PID.)

The MPC tab tunes the horizon N, tracking weight Q, and move-suppression R live.

Auto-calibration (safe, model-based)

The calibration discovers the valve deadband and proposes PID gains without ever risking the hardware. The textbook autotuners — Ziegler–Nichols and relay feedback — deliberately drive the closed loop to (or into) sustained oscillation to find the stability limit; on a heavy hydraulic axis that is exactly what you don't want. Instead this runs a short, scripted set of gentle, bounded, open-loop moves and computes the gains from the resulting model — nothing is ever driven unstable.

The identification experiment

It runs live as a Controller state machine, so you watch it happen:

  1. Park — ease the rod to a safe start (~20% stroke) with a PWM-capped proportional approach.
  2. Deadband ramp — from rest, slowly raise the PWM from zero until the rod begins to move, recording the (PWM, velocity) pairs as it starts.
  3. Settle — command zero and let it stop.
  4. Step — apply one bounded constant-PWM step and record the velocity response.
  5. Return — ease back toward the park position and finish.

Throughout, the PWM is capped and the position is kept inside a safe window (the run aborts if it would leave it), with per-phase watchdog timeouts.

From data to model

The deadband is found by back-extrapolation: above it, velocity rises roughly linearly with PWM, so the rising (PWM, velocity) points are fitted to a line and the deadband is the PWM where that line crosses zero velocity. This removes the bias a plain "first motion" threshold suffers from the valve lag and the ramp speed (reading ~9% vs. the true 8%, instead of ~12%). The step gives the velocity gain Kv = v_ss / (step PWM − deadband) (mm/s per %) and the lag τ (time to reach 63% of the steady velocity).

From model to gains (SIMC / lambda)

The cylinder behaves as an integrating-plus-lag process P(s) = Kv / (s(τ s + 1)) — PWM sets a velocity that integrates to position. The gains follow from SIMC (Skogestad's internal-model-control tuning) for that process, whose single knob is the desired closed-loop time constant τc — the aggressiveness control (larger = gentler): 

theta = tau                  # treat the lag as an effective delay
tc    = max(tau_c, theta)    # never ask to be faster than the lag
Kc    = 1 / (Kv * (tc + theta))
tauI  = 4 * (tc + theta)
tauD  = tau

kp = Kc           # %/mm
ki = Kc / tauI    # %/(mm·s)
kd = Kc * tauD    # %/(mm/s)

The presets map Conservative / Balanced / Aggressive to τc = 0.5 / 0.3 / 0.15 s. The result includes a modest I term, which is what lets the tuned PID reject the load and the deadband offset.

Working at any load

The deadband and velocity gain are intrinsic actuator properties — a constant load only adds a force offset, it doesn't change them. So the calibration temporarily lifts the load for the duration of the run (the drawn weight shows lifted): the identification stays clean and the experiment's bounded PWM is always enough to move the rod, whatever the load slider says. The load is restored when the run finishes, is interrupted, or on Reset; the tuned PID then handles the real load through its integral.

When the run completes, the tab shows the deadband, gain, lag, and proposed gains; Apply writes them into the PID tab and switches to it.

8. Build & run instructions

No compiler needed — it's pure Python.

# 1. Create and activate a virtual environment (Windows PowerShell)
python -m venv .venv
.\.venv\Scripts\Activate.ps1

# 2. Install dependencies
pip install -r requirements.txt

# 3. Run
python main.py

requirements.txt:

PySide6
pyqtgraph
numpy

Later, a one-file Windows executable can be produced with pyinstaller --noconsole --onefile main.py.
